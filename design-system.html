Design System

AAN Master 2026

A token-first system. Every color, size, radius, font, shadow and motion value is defined once in aan-theme-kit.scss and consumed everywhere else. Phase 2 establishes the typography and rhythm foundations; components arrive next.

Default master ships neutral (premium comes from type + spacing, not a brand hue); clones set $color-primary — see Aurum in the skin switcher for an example.

Find any component with the sidebar search (name or keyword). Specimens are fluid (they respond to the page width) unless a Desk / Mobile toggle is shown — those wrap a fixed frame so you can preview the component's mobile state on the spot.

Principles

1. Tokens are the single source of truth — no magic numbers in partials.

2. Bootstrap supplies grid, utilities and JS only; visuals route through the bridge.

3. Section rhythm is deliberate — bottom padding is always larger than top.

4. Nothing "AI-default": no cream+terracotta, no acid green, no gradient buttons.

5. Hero has two contracted modes — full (no fold bleed) and compact (next section visible above the fold).

6. Accessibility is a system contract, not a nice-to-have (see below).

WCAG AA is a system contract, not a nice-to-have.

All text/background token pairs pass AA contrast (4.5:1 small, 3:1 large); every interactive element has a visible :focus-visible ring; keyboard and reduced-motion paths are first-class. Clones changing colors MUST re-verify contrast — because color lives in the skin map, the token structure makes this a one-file check (the live ratios in Foundations re-compute per skin).

Do

Add a token before you reach for a literal value.

Compose rhythm from $section-* and $stack-* gaps.

Keep the DS chrome quiet — let specimens lead.

Don't

Hard-code hex, px, or durations outside the token file.

Restyle a Bootstrap component inline or in an ad-hoc file.

Introduce components or pages the plan didn't ask for.

Transparent-at-top contract

The real .site-header is position: fixed and overlays the hero. At scrollY == 0 it wears .site-header--top: fully transparent (no bg / blur / border), logo + links + hamburger render in var(--header-top-fg) — a per-skin token (light over a photo hero; $color-heading over gallery's plain hero) so it's always legible.

Hero fold contract

.hero--full: fills the viewport under the sticky header — height: calc(100svh - $header-h) (fallback 100vh); content vertically centered, spec-strip pinned to the fold bottom, safe-area bottom padding; short landscape (<560px tall) falls back to min-height so nothing clips. NOTHING below it may be visible before scroll — zero bleed at the fold.

.hero--compact: intentionally shorter — height: clamp(480px, 62svh, 640px); the next section MUST be visibly peeking above the fold (on 1080p desktop and common mobile heights). If the next section isn't visible, the height values are wrong — fix tokens, not content.

Content: BOTH modes center the text block (eyebrow/h1/lead/CTAs) vertically as a group, with even air above and below. Full pins the spec-strip to the hero bottom (out of the centering flow); compact HIDES the spec-strip and caps its height (smaller h1 clamp) so the reduced set — both CTAs included — sits fully above the fold with the next section peeking. Compact keeps the next section visible via the capped height, not a top anchor.

Both modes work with both hero styles — the full × compact × photo × plain 2×2 matrix is verified. Try the Hero · Full / Compact toggle (above the skin switcher) on the homepage, and switch skins to flip photo ↔ plain (see hero-style).
