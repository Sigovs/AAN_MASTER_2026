# AAN Master 2026 — Project Rules (CLAUDE.md)

These rules are **permanent** and apply to every file in this repository. They are
not suggestions. If a change would violate one of them, stop and reconsider the
approach instead of working around the rule.

> Source of authority: `AAN-DESIGN-SYSTEM-PLAN.md` (the master brief). When this
> file and the plan disagree, the plan wins — update this file to match.

## 0.9 DS documentation contract

Every global addition or change — a new token, component, pattern, modifier, or
a changed rule (rhythm, fold contract, usage constraint) — MUST land on
`design-system.html` in the same run: a specimen or tile, a class-name caption,
and a `.ds-usage` note where behavior rules exist. A build that adds a global
capability without its DS entry is incomplete, even if it compiles green. When
in doubt whether something is "global": **if a clone could use it, it is.**

## 1. Tokens are the single source of truth

- **Every** color, spacing value, border-radius, font-family, font-size,
  shadow, transition, and z-index comes from a token defined in
  [`src/scss/aan-theme-kit.scss`](src/scss/aan-theme-kit.scss).
- **No magic numbers anywhere else.** No hard-coded hex colors, no `16px`,
  no `0.3s`, no `999` z-index literals in any partial, component, or HTML
  `style` attribute. If you need a value, add a token for it first.
- If a value is used more than once, or is likely to change, it is a token.

## 2. Token naming — AAN Master 2026 Theme Kit

Token names follow the AAN Master 2026 Theme Kit convention (plan §1). Examples:

| Concern      | Example token     |
| ------------ | ----------------- |
| Brand color  | `$color-primary`  |
| Page canvas  | `$bg-page`        |
| Card surface | `$surface-card`   |
| Display font | `$font-display`   |

Keep the existing names. Do not rename tokens to personal preference — the names
are a contract with the design side.

## 3. Bootstrap 4 is structure only

- Bootstrap 4 provides **grid, utilities, and JS behavior only**.
- **Prototype targets BS4 to match production; BS5 upgrade is a separate future
  recommendation.** Production runs Bootstrap 4.0 (JS self-hosted, CSS compiled
  from their Sass); the prototype pins **Bootstrap 4.6.2** via CDN. Use BS4
  utility naming (`.mr-*` / `.ml-*`, not BS5's `.me-*` / `.ms-*`).
- **All visual styling** (color, type, borders, shadows, component looks) is
  applied through [`src/scss/bootstrap-bridge.scss`](src/scss/bootstrap-bridge.scss),
  which maps our tokens onto Bootstrap's components (BS4 has no `--bs-*` custom
  properties, so the bridge sets explicit properties).
- Never restyle a Bootstrap component inline or in an ad-hoc partial. Route it
  through the bridge so the mapping stays in one place.

## 4. Forbidden — "AI default" looks

Do not produce, and actively avoid, the visual clichés below. They are banned
regardless of how convenient they seem:

- **Cream + terracotta** palettes.
- **Black + acid/neon green** palettes.
- **Gradient buttons** (or gradient-filled primary CTAs generally).
- Anything **not derived from the plan** — no invented components, no invented
  pages, no decorative flourishes the brief did not ask for.

Build only what the plan specifies.

## 5. Section rhythm

- A section's **bottom padding is always larger than its top padding.** This is
  a rhythm rule, not a preference. Section padding tokens are defined as pairs in
  `aan-theme-kit.scss`; the bottom value in every pair is the larger one.

## 5b. Hero fold contract

The hero has two contracted height modes. **`.hero--full`** occupies the entire
viewport under the sticky header — `height: calc(100svh - $header-h)` (fallback
`100vh`); content is vertically centered, the spec-strip is pinned to the fold
bottom (`margin-top:auto`), with safe-area bottom padding; on short landscape
(height < 560px) it falls back to `min-height` so content never clips. NOTHING
below it may be visible before scroll — zero bleed at the fold. **`.hero--compact`**
is intentionally shorter — `height: clamp(480px, 62svh, 640px)`; the next section
MUST be visibly peeking above the fold (on 1080p desktop and common mobile
heights). If the next section isn't visible, the height values are wrong — fix
the tokens/height values, not the content. Both modes must work with both hero
styles (photo / plain).

**Content contract:** BOTH modes center the text block (eyebrow/h1/lead/CTAs)
vertically as a group, with even air above and below. `.hero--full` centers it in
a 1fr grid row and pins the spec-strip to the hero bottom (separate `auto` row, so
it does not affect the centering). `.hero--compact` HIDES the spec-strip and caps
the height (and may reduce the display size — smaller h1 clamp) so the whole
centered set — including both CTAs — sits fully above the fold with the next
section still peeking. Compact keeps the next section visible via its capped
height, NOT a top anchor.

## 5c. Accessibility is a contract

WCAG AA is a system contract, not a nice-to-have: all text/background token
pairs pass AA contrast (4.5:1 small, 3:1 large), every interactive element shows
a visible `:focus-visible` ring, and keyboard + reduced-motion paths are
first-class — any clone that changes colors MUST re-verify contrast (a one-file
check, because color lives in the skin map; the DS Foundations section computes
the live ratios per skin).

## 5d. Mobile media rule

Bleed / feature imagery (`.media-split`, `.media-duo`, `.showcase`) is **NEVER
hidden on mobile — it stacks.** Hiding imagery to "simplify" mobile is forbidden.
If an image genuinely can't work at mobile widths, that is a design problem to
solve, not an element to `display:none`. (Watch for the inverse failure too: a
desktop grid-column placement leaking into a single-column mobile layout can
starve an image track to `0px` — the image is "gone" without a `display:none`.
Lay mobile stacks out so column placement can't leak in.)

## 6. Working style

- **Never ask yes/no or confirmation questions; never pause for permission
  between steps.** Make senior-level decisions and continue, flagging judgment
  calls in the final report only. Sole exception: external blockers Claude
  cannot resolve alone (auth prompts, locked files, missing credentials) — stop
  and state exactly what the user must do.
- **Run `npm run build` before committing.** `dist/css/main.css` is committed by
  design so the repo is viewable (GitHub / Pages) without a build step; other
  build junk stays ignored.
- Phase discipline: do only the current phase's work.
- Prefer editing a token over adding an exception.
- Keep the SCSS import graph flowing through `main.scss`.
